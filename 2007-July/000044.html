The new `-force-restore' feature works well. It allowed me to restore my old 
dumps (that I made for testing) properly.

However, the second new feature is not yet functioning correctly. Empty dumps 
*without* a volume header give

  $ star t f= level-1.star 
  ./ directory link to .

Restoring such an empty dump gives

  Validating this dump against restored filesystem...
  Dump is valid, starting restore.
  star: Archive contains directory './' without name list

The exit code is 255. Attempts to restore higher-level dumps fail:

  star: File exists. Cannot create 'star-lock', restore is already running.

Deleting the lock file by hand and continuing using `-force-restore' works.

Empty dumps *including* a volume header show the old behaviour; there seems 
to be no `.' in them.
